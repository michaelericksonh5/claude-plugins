# Spine CLI Reference

This reference captures the official Spine command-line families the skill should understand. It does not expand what the plugin is allowed to automate. The plugin only automates policy-approved CLI paths; production export remains gated.

## Plugin Support Status

| Status | CLI use | Plugin boundary |
|---|---|---|
| Supported / automated | Import a generated `.json` package into a native `.spine` project with `-i -o -r`; probe project info with `-i`. | Used by `scripts/create_spine_project.py`, `--create-spine-project`, and the handoff ZIP script on machines with an approved licensed Spine CLI. |
| Documented but policy-gated | Export to JSON, binary / `.skel`, export with packing, clean, pack, unpack, and advanced flags. | The skill may explain the official commands, but production export remains gated and must not be invoked without explicit approval. |
| Unsupported / not automated | License activation beyond logout, Perforce submission, source-art writes, and production export without approval. | Do not automate these paths. Do not write to Perforce or source-art roots. |

## Editor And General Commands

The general command family includes:

- `-h` / `--help` for command help.
- `--advanced` for advanced command help.
- `-v` / `--version` for version output.
- `-u` / `--update` with optional `-f` / `--force` for updating Spine.
- `-l` / `--logout` for logging out.
- `-x` / `--proxy` for proxy configuration.
- `-t` / `--notimeout` for disabling timeout behavior.
- An optional project path to open a project.

Plugin status: help/version/update/logout/proxy details are reference-only. The plugin does not automate license activation, and logout is the only license-account command documented here as an official CLI action.

## Export

Official shape:

```text
Spine [-i <path>] [-m] [-o <path>] -e <path>
Spine [-i <path>] [-m] [-o <path>] -e json[+pack]|binary[+pack]
```

Inputs may be a folder, project, or data file. Outputs may be a file or folder. `--clean` can clean animations before export, and advanced `--clean-all` applies cleanup to all exports.

Plugin status: documented but policy-gated. The production export gate is non-exporting; it can report `ready_for_manual_export`, but it does not invoke Spine CLI export, create `.skel`, or write production files. Production export remains gated.

Autonomous-rigger proposals do not change this boundary. `scripts/generate_rig_proposal.py` is report-only, and even a clean proposal cannot enable `.skel` or JSON production export automation. An approved policy must still name exact editor/runtime versions, output format, license posture, runtime evidence, and approver metadata before any manual export readiness claim.

## Import

Official shape:

```text
Spine -i <path> [-s <scale>] -o <path> -r [<name>]
```

Inputs may be a folder, project, JSON data, or binary data. Output is a Spine project. `-s` optionally scales import data, and `-r` optionally renames the imported skeleton.

Plugin status: supported for generated package JSON to native `.spine` creation. The approved wrapper uses the official import path, typically:

```powershell
Spine -i <compiled_package_dir>\shared_symbols.json -o <output_project.spine> -r <skeleton_name>
```

Import data intended to round-trip back to the editor should come from JSON or binary exported with **Nonessential data** checked when editor fidelity matters. Mesh edges are nonessential editor data.

## Clean

Official shape:

```text
Spine -i <path> -m
```

This opens a project or data input, cleans animations, and saves the project.

Plugin status: documented but policy-gated. The plugin does not run cleanup against artist project files.

## Pack

Official shape:

```text
Spine -i <path> [-j <path>]... -o <path> -p <name|pack-settings.json>
```

Optional `-n <name>` controls atlas naming. Optional project references can be supplied to strip mesh whitespace during packing.

Plugin status: documented but policy-gated. The plugin has its own deterministic atlas builder for proof packages and does not use official pack for production writes.

## Unpack

Official shape:

```text
Spine -i <path> -o <path> -c <atlas>
```

This unpacks an atlas into images.

Plugin status: documented but policy-gated. Handoff ZIPs include unpacked `editor_images/` generated by plugin tooling; they do not require production atlas unpack automation.

## Info

Official shape:

```text
Spine -i <path>
```

This reports information for the input project or data.

Plugin status: supported as a project probe after native `.spine` creation. `scripts/create_spine_project.py` uses it to verify the generated project when probing is enabled.

## Advanced Flags

Official advanced flags include:

- JVM memory: `-Xmx`.
- Diagnostics and server behavior: `--trace`, `--auto-start`, `--no-auto-start`, `--ping`, `--server`.
- Runtime/editor behavior: `--disable-audio`, `--pretty-settings`, `--keys`, `--hide-license`, `--ui-scale`, `--icc-profile`, `--intro`.
- Export and import behavior: `--clean-all`, `--mesh-debug`, `--export-selection`, `--ignore-unknown`.

Plugin status: documented but policy-gated. Do not use advanced flags to bypass the production export gate, source-art write rules, Perforce rules, or license policy.

## Platforms

- Windows: prefer `Spine.com` for CLI automation because it waits for completion and emits console output. `Spine.exe` is the GUI launcher.
- macOS: invoke the executable inside the Spine app bundle.
- Linux: invoke `Spine.sh`.

## Invocation Semantics

Multiple commands can be specified in one invocation. Output folders are created as needed. A non-zero exit code means the CLI command failed and must block the plugin workflow.

The plugin should surface CLI failures directly in reports instead of treating partial output as success.
